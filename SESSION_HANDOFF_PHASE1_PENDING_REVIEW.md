### Pending — follow-up ticket
Project lead to manually file (GitHub API write permission not available from this environment):

**Title:** OCR cache key should include Document AI configuration to prevent stale-cache hits across processor changes

**References:** Migration #061, ADR-025, `server/services/ocr-cache.ts`, `server/routes/ai/extraction.ts:2065`

**Body summary:** The OCR cache table (Migration #061) keys entries by `sha256(sourceBytes):chunk/total` only. No processor ID, project ID, or model version is included in the cache key or schema. ADR-025 acknowledges this as accepted v1 risk with manual purge as the only mitigation. Investigation surfaced this risk is broader than originally captured: any consumer hitting the cache after a Document AI configuration change receives text generated by the prior configuration with no detection mechanism, no metadata indicating which processor generated the cached text, and no automatic invalidation. The bypass guard added in this session expands the surface where stale cache entries can be served (uncredentialed environments can now serve cached text from a credentialed-environment-generated cache, with no live OCR call to fail and surface configuration drift).

### Pending — D-prime decision
A separate consideration emerged: the credential check at `/api/ai/ocr/document-ai` runs before the cache lookup, meaning cache hits cannot be served when credentials are missing even though they don't strictly require live credentials. The ordering was found to be incidental rather than deliberate. The fix (move cache lookup before credential check) was investigated but not applied — project lead chose to provide credentials rather than modify the production route. The reordering remains a candidate for a focused PR if and when project priorities allow, ideally bundled with the OCR cache config-awareness ticket above so both cache concerns are addressed together.

# ADR 0011: Display Interpreter & Validation Engine

## Status
Accepted

## Date
March 15, 2026

## Context
In early versions of the InsurAI platform, raw JSON extractions generated by the LLM (GPT-4o or Claude 3.5) were persisted directly to the database and passed relatively unchanged to the front-end components for display.

While this allowed for fast iteration, it presented several unacceptable risks for a production insurance tool:
1. **Forbidden Phrase Leakage:** The LLM could occasionally hallucinate or extract prohibited marketing phrases like "sınırsız" (unlimited) for limits that are actually legally capped (e.g., İMM limits). Displaying these to end-users exposes the platform to legal and compliance risks.
2. **Contradiction Propagation:** The LLM might extract a "conditional deductible" as a boolean `true` while simultaneously failing to extract the actual condition text, leading to a contradictory and confusing UI state.
3. **Over-Confidence:** Lower confidence extractions or OCR-garbled text inputs could still result in a complete-looking UI, masking the risk that the AI missed critical coverage exclusions.
4. **Lack of Human Review Gates:** There was no structural way to intercept borderline extractions and force them into a "Human Review Required" workflow before showing them to the user.

## Decision
We implemented a strict, multi-layer validation and display interpretation pipeline to completely decouple raw AI extractions from the UI display layer.

This pipeline consists of:
1. **Document Normalizer (`document-normalizer.ts`)**: Applies deterministic pre-processing to the raw LLM output, filling in missing defaults and conforming the payload to a strict type.
2. **Validator Engine (`validator.ts`)**: Evaluates the normalized payload against branch-specific logical rules (e.g., condition-contradiction detection, missing critical fields) and generates a list of "warning flags".
3. **Review Threshold Logic (`review-thresholds.ts`)**: Synthesizes the LLM's self-reported confidence scores along with the validator's warning flags to assign an immutable `displayMode` (`full`, `restricted`, or `human_review_required`).
4. **Display Interpreter (`display-interpreter.ts`)**: The final gatekeeper. It consumes the validated payload and the `displayMode`, scrubs the text arrays (cleansing prohibited phrases), masks omitted or blocked fields (based on the restricted mode), and outputs a `DisplaySafePolicySummary`.

*Only the `DisplaySafePolicySummary` is permitted to be consumed by UI components like `PolicyCard` and `PolicyDetailView`.*

## Consequences

**Positive:**
- Complete elimination of prohibited marketing phrases from the UI.
- Deterministic, testable boundaries that prevent edge-case hallucinations from reaching the end user.
- Ability to enforce a safe rollout strategy (e.g., KASKO internal pilot) by hooking into the `human_review_required` pathway.
- Simplification of React components, which no longer need to check for raw nulls, missing fields, or contradictory states.

**Negative:**
- Increased pipeline complexity; adding a new coverage or feature now often requires updating the AI prompt, the JSON schema, the normalizer, the validator, and the display interpreter.
- Slower end-to-end extraction processing time (marginally increased due to synchronous validation looping, though still under ~20ms CPU time post-LLM fetch). 
- Legitimate, highly-confident extractions on poorly scanned documents may now be artificially degraded to `restricted` mode, requiring more manual intervention by users or admin reviewers.
